Reference for every field in a MekariRAG knowledge base article. Use this page to understand what value to put in each field when reviewing or editing.

Article Type

Determines the article's purpose and which sections are required. MekariRAG's retriever uses this to set expectations about what the chunk will contain.

task 278 articles

A step-by-step guide for completing a specific action in the product. Use this when the user wants to do something: set up a feature, run a report, configure a setting.

Required sections: Prerequisites → Steps → Expected Result → Error States → Escalation

Most articles are tasks. If the user's question starts with "Bagaimana cara…" it's almost always a task.
concept 86 articles

An explanatory article about a product area, feature, or idea — no hands-on steps required. Use this for overview pages, definition pages, and "what is X" content.

Required sections: Definition → Why It Matters → Key Attributes → Related Tasks

If the article is mostly explanation with no numbered steps, it's a concept.
troubleshooting 3 articles

A guide for diagnosing and fixing a specific problem. Use this only when the article is primarily reactive — the user already has a broken or unexpected state and needs to resolve it.

Required sections: Symptom → Root Cause → Solution → Escalation

Rare in this KB. Most "error" content lives in the Error States section of a task article instead.

Solvability Type

Tells MekariRAG how the user can resolve their need — whether by using the product, reading content, or a mix of both. Affects how the AI bot frames its answer.

tool 253 articles

The user can solve their need themselves by taking an action inside the product. There is a button to click, a form to fill, a setting to change.

Example: "Cara menambahkan team member" — the user goes to Settings and clicks a button.

content 86 articles

The user's need is met by reading or understanding the content. No product action required. Typically paired with article_type: concept.

Example: "Overview dashboard supervisor" — the user just needs to understand what they're looking at.

hybrid 28 articles

Part of the answer is in the product (an action to take), and part requires understanding a concept or policy first. Use when the article combines explanation and steps.

Example: "Cara mengatur approval limit" — first explains what approval limits are, then shows how to configure them.

Product Surface

Where the user performs the action described in the article. Helps the AI bot give context-appropriate answers (e.g., "open the mobile app" vs "go to the web dashboard").

webDesktop browser (299 articles)
mobileiOS / Android app (49 articles)
apiREST API / developer integration (14 articles)
allApplies to all surfaces (5 articles)
If an article covers both web and mobile, pick the primary surface and note the other in the content. Only use all when the concept truly applies everywhere (e.g., billing, account overview).

Confidence Level

How reliable the content in a section (or the whole article) is. Set by the AI during enrichment and adjustable during review. MekariRAG uses the article-level value as faithfulness_threshold to decide how strictly to verify AI answers against this source.

high ✓ faithfulness_threshold: 0.9

Content was directly extracted from the Zendesk source and is factually grounded. The AI generated it from clear, explicit source material. MekariRAG will require a 90% faithfulness match before using this source in an answer.

Good for: Prerequisites, Steps with clear UI references, Expected Result when the outcome is obvious.

medium ~ faithfulness_threshold: 0.6

Content was inferred or reconstructed from partial source material. The AI had to make reasonable assumptions to fill the section. Human review recommended before relying on this in production.

Common for: Escalation paths, general error guidance not explicitly stated in the source.

low ? faithfulness_threshold: 0.3

Content is largely invented by the AI — the source article did not contain enough information to fill this section. Treat this as a placeholder. A human must verify or rewrite before the article is production-ready.

Common for: Error States when no errors were documented, or any section where the source was missing.

Low-confidence articles are flagged in the review table. Prioritise these for human review.
The article-level confidence (shown in the sidebar) is the lowest section confidence in that article. If any one section is low, the whole article is low.

Section Structure

Each article type has a fixed set of required H2 sections. MekariRAG splits articles on H2 headings — each section becomes a retrievable chunk. Sections must be self-sufficient: the AI bot may retrieve only one section at a time, so each must make sense without reading the others.

task

SectionWhat it should answer
PrerequisitesWhat does the user need before they start? Role, permissions, active features, required data.
StepsExact numbered actions in the UI. Each step = one action. Include button names, menu paths, field names.
Expected ResultWhat does success look like? What changes in the system, what confirmation does the user see?
Error StatesWhat can go wrong? List specific errors with their solution. If none are known, say so honestly.
EscalationWhen should the user contact support? What information should they include?

concept

SectionWhat it should answer
DefinitionWhat is this feature/term in plain language? One clear sentence, then expand.
Why It MattersWhy would a user care? What business problem does this solve?
Key AttributesImportant properties, limits, or behaviours. Good for tables or bullet lists.
Related TasksLinks or references to task articles that put this concept into action.

troubleshooting

SectionWhat it should answer
SymptomExactly what the user sees or experiences. Specific error message, wrong behaviour, missing feature.
Root CauseWhy it happens. May be a configuration issue, permission gap, or known limitation.
SolutionStep-by-step fix. If multiple causes → multiple solutions.
EscalationWhen the fix doesn't work — who to call and what to bring.
Golden rule: every section must contain the article title, product name, and context. Do not write "Click the button" — write "Klik tombol Buat Team di halaman Team Settings Qontak CRM."

Intent Tags

Short, hyphenated labels that describe what the user wants to do when they would find this article useful. MekariRAG uses these to match ambiguous queries to the right article.

Format: lowercase, hyphen-separated, action-oriented. Examples: add-team-member, integrate-instagram, reset-password

How many: 3–6 tags per article. More is not better — be specific.

Avoid: product names as tags (that's what products is for), article type words like "task" or "guide", vague tags like "settings" alone.

Ask: "What would a support agent type to find this article?" Those keywords become your intent tags.

Query Examples

Sample natural-language questions a user might type into the AI bot that this article should answer. Used to train and test the retriever.

Format: Full sentences or questions. Mix formal and casual Bahasa Indonesia. Include at least one English variant.

Examples:

  • "Bagaimana cara menambahkan team di Qontak CRM?"
  • "Cara buat team baru di CRM gimana?"
  • "How do I add a team in Qontak?"

How many: 3–6 examples. Cover different phrasings of the same intent.

These examples directly influence retrieval quality. Bad examples = the bot can't find the article when it should.

Compliance Sensitive

Flags articles that touch legally or regulatorily sensitive topics (data privacy, financial transactions, user data deletion, export controls, etc.). MekariRAG may apply stricter answer validation for these articles.

falseStandard article — no special handling
trueCompliance-sensitive — human review required
Mark true if the article covers: personal data handling, consent or opt-out flows, billing or payment actions, account deletion, legal agreements, or regulatory deadlines (e.g. WhatsApp policy compliance, PDPA, OJK rules for fintech integrations).

Plan Scope

Restricts which subscription plans this article applies to. If the feature is available on all plans, leave this null.

Format: A string describing the plan restriction. Examples: "Enterprise only", "Starter and above", "Add-on required"

When set, MekariRAG can prepend a plan-gate notice to the AI answer so users on lower plans aren't sent down a path that doesn't apply to them.

Most articles: null.

Chunk Groups & Related Chunks

Advanced metadata used to link related articles together in the retriever. Leave empty for now — these are populated after the full KB is indexed.

chunk_groups

A list of group labels that cluster articles by topic area. When a query matches one article in a group, MekariRAG can fetch others in the same group to give a more complete answer.

Example: ["whatsapp-setup", "channel-integration"]

related_chunks

A list of article slugs that are closely related and should be surfaced together. Used for "see also" linking at the retrieval layer.

Example: ["5536149093273-bagaimana-cara-mengintegrasikan-instagram", "5521732886553-bagaimana-cara-mengintegrasikan-facebook"]

Verified By / Last Verified

Tracks who last confirmed the article's accuracy and when. Used to surface stale content for re-review.

verified_by

The reviewer's identifier. Current value for AI-enriched articles: AI:claude-haiku-4-5-20251001.

When a human reviews and approves, update this to their name or email, e.g. budi@mekari.com.

last_verified

ISO date of last verification. Format: YYYY-MM-DD. Example: 2026-04-18.

Articles not verified within 6 months should be flagged for re-review, especially for features that update frequently (e.g. WhatsApp API, chatbot builder).
